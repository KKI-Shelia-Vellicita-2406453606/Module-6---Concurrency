## Commit 1 Reflection Notes 

The 'handle_connection' function utilizes a BufReader to buffer the raw TcpStream in memory, which is a common pattern in Rust for handling I/O efficiently. This approach significantly improves performance by reducing the frequency of direct, expensive system calls to the operating system. By employing 'take_while(|line| !line.is_empty())', the code accurately recognizes the blank line that signals the end of the HTTP headers in a standard request. This specific stop condition is essential because the HTTP protocol dictates that headers are always followed by a CRLF sequence. Without this check, the server would wait indefinitely for more data that the browser isn't yet sending. This results in a deadlock where the browser expects a response while the server remains stuck in an infinite reading loop.

## Commit 2 Reflection Notes

The 'handle_connection' function was updated to return a functional HTTP response by reading the hello.html file using the 'fs::read_to_string' method. To comply with the strict HTTP/1.1 protocol standards, a response string is constructed using the format! macro to organize the metadata. This string includes a standard status line of HTTP/1.1 200 OK and a mandatory Content-Length header so the browser knows how much data to expect. The crucial double line break (\r\n\r\n) is also included to act as a structural delimiter between the headers and the actual HTML body. This formatted string is then converted into a byte slice via as_bytes() because network streams communicate in raw bytes rather than high level characters. Finally, the data is sent through 'stream.write_all', ensuring the browser receives all necessary information to correctly interpret and render the payload.

## Commit 3 Reflection Notes

The 'handle_connection' function was refactored to introduce basic routing logic by validating the initial request line sent by the client. By utilizing 'buf_reader.lines().next()', the server efficiently isolates the specific HTTP method and path requested by the browser. This approach allows the request's intent to be determined immediately without needing to parse the entire header block unnecessarily. The conditional branching was implemented where the server checks if the request line matches the specific root path of GET / HTTP/1.1. If the match is successful, the server returns a successful status along with the content of hello.html. Otherwise, it defaults to an HTTP/1.1 404 NOT FOUND status serving a custom 404.html file to inform the user the resource is missing.

## Commit 4 Reflection Notes

The 'handle_connection' function was further refactored to eliminate code duplication by utilizing a match expression for much cleaner logic. This expression assigns the status_line and filename dynamically based on the validated request line, which keeps the response-sending code DRY (Don't Repeat Yourself). The /sleep route is also introduced using 'thread::sleep(Duration::from_secs(10))' to simulate a slow, high latency processing task. This experiment revealed a critical performance bottleneck because the current server implementation is strictly single threaded. Because the server can only process one request at a time, a long running request effectively prevents all other users from accessing the server until that task completes. This exercise highlights the absolute necessity of concurrency (such as a thread pool) to handle multiple simultaneous connections efficiently.

## Commit 5 Reflection Notes

In this milestone, the server's resource management is significantly improved by implementing a graceful shutdown mechanism through the Drop trait. The mpsc::Sender is wrapped in an Option so that it could be explicitly dropped during the shutdown process to signal the workers. Dropping the sender informs the worker threads that the channel is closed and no more incoming jobs will be processed. The worker threads now utilize a match expression on the recv() result to detect this closure and break their internal loops safely. In main.rs, the use of .take(2) on the connection listener provides a controlled environment to verify that the shutdown sequence works as intended. This ensures that the ThreadPool is dropped and all worker threads are joined correctly, preventing "zombie" processes and ensuring a clean exit for the application.

## Commit 6 Reflection Notes (Bonus)

The introduction of the 'build' function serves as a more idiomatic approach to object instantiation in Rust when there is a potential for failure. Unlike the standard new function, which relies on a hard 'assert!' that triggers a global panic if the input is invalid, build returns a Result type that wraps the ThreadPool. This architectural shift allows the calling code in main.rs to decide exactly how to handle a configuration error, such as logging a custom error message or exiting with a specific status code rather than allowing the entire application to crash unexpectedly. In the context of a web server, providing a graceful exit or a fallback configuration is much safer than a panic, especially in production environments where uptime is critical. Furthermore, the build pattern follows the 'fail fast but fail gracefully' philosophy, making the API more expressive and communicative about its internal constraints to other developers. By comparing the two, it becomes clear that while new is convenient for internal logic where success is guaranteed, build offers superior robustness and gives the developer total control over the application's overall lifecycle.
